# 09. Pilot Day Runbook

개발자 없이도 현장에서 파일럿을 실행할 수 있도록, 당일 운영 절차를 순서대로 정의한다.

## 역할 정의

- 운영 담당자(매장): 실제 통화 발생/업무 진행, 앱 화면 확인
- 파일럿 리더(본사/관리자): 체크리스트 진행, 결과 승인, 이슈 분류
- 서버 모니터 담당(원격 가능): API 응답/처리 로그 확인, 장애 여부 판단

## 사전 준비 (시작 30분 전)

1. 앱 실행 및 로그인 상태 확인
   - 확인자: 운영 담당자
   - 성공 기준: `Login` 상태 성공, `Settings`에서 폴더 URI 표시
   - 문제 대응: 로그인 실패 시 계정 확인 -> 재로그인 -> 계속 실패 시 파일럿 리더 호출

2. 대상 폴더 권한 확인(SAF)
   - 확인자: 운영 담당자
   - 성공 기준: 녹음 폴더명/URI가 보이고 재실행 후에도 유지
   - 문제 대응: 폴더 재선택 -> 권한 재부여 -> 여전히 실패 시 기기 재부팅 후 재시도

3. 서버 연결 확인
   - 확인자: 서버 모니터 담당
   - 성공 기준: `/api/mobile` 로그인/업로드 URL API 응답 정상(401/5xx 없음)
   - 문제 대응: 서버 장애 분리(앱 문제 아님) 후 장애 공지, 복구 시점 재개

## 파일럿 본 실행 (운영 시간)

### 단계 A: 기준 통화 1건 검증

1. 테스트 통화 1건 발생
   - 확인자: 운영 담당자
2. `Run Step 2: Scan Worker` 실행
   - 확인자: 운영 담당자
   - 성공 기준: scan 로그에 `inserted >= 1`
3. `Run Step 3: Upload Worker` 실행
   - 확인자: 운영 담당자
   - 성공 기준: upload 로그에 `uploaded >= 1`
4. `Status` 화면 확인
   - 확인자: 파일럿 리더
   - 성공 기준: 상태 전이 `pending -> uploading -> uploaded -> synced`, `serverCallId` 표시
5. 서버 call 생성 확인
   - 확인자: 서버 모니터 담당
   - 성공 기준: call row 생성 및 중복 없음

장애 대응:
- `retry_pending`이면 네트워크/토큰 상태 먼저 확인 후 재실행
- `failed_upload`이면 즉시 파일럿 리더에게 보고, 에러메시지와 시각 기록

### 단계 B: 실제 운영 관찰 (최소 1~2시간)

1. 실제 통화 흐름 유지
   - 확인자: 운영 담당자
2. 15~30분 간격으로 `Status` 확인
   - 확인자: 파일럿 리더
   - 성공 기준: 실패 건이 누적되지 않고 synced 비율 증가
3. 실패 필터 점검
   - 확인자: 파일럿 리더
   - 성공 기준: 실패/재시도 대상이 명확히 보이고 원인 파악 가능

장애 대응 우선순위:
1) 네트워크 단절 여부
2) 토큰 만료/재로그인 필요 여부
3) SAF 폴더 권한 손실 여부
4) 서버 장애(401/5xx 급증) 여부

## 종료 점검 (마감 전 15분)

1. `Status` 필터별 건수 집계
   - 확인자: 파일럿 리더
   - 성공 기준: 미처리(`retry_pending`, `failed_upload`) 건수와 원인 파악 완료

2. 파일럿 기록 양식 작성 완료
   - 확인자: 운영 담당자 + 파일럿 리더
   - 성공 기준: 필수 항목 누락 없음

3. 이슈 분류 및 후속 액션
   - 확인자: 파일럿 리더
   - 분류: 앱/서버/네트워크/운영절차
   - 후속: 다음 파일럿 전 수정 항목 확정

## 당일 성공 판정 기준

- 기준 통화 1건이 end-to-end `synced`까지 도달
- 전체 통화 대비 call 생성 누락이 없음(또는 원인 설명 가능)
- 치명 장애(앱 크래시, 데이터 유실, 무한 실패)가 없음
- 실패 건은 모두 원인 코드(`AUTH_EXPIRED`, `RELOGIN_REQUIRED`, 네트워크 등)로 분류 가능
